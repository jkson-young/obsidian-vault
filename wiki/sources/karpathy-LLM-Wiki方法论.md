# karpathy LLM Wiki 方法论

> 原始资料：[[karpathy wiki方法论]]
> 作者：Andrej Karpathy

## 核心思想

区别于传统 RAG（每次查询重新检索片段），LLM Wiki 的核心理念是：

> **增量构建并维护持久的 wiki** —— 一个结构化的、相互链接的 Markdown 文件集合，位于用户和原始资料之间。

### 与 RAG 的关键区别

| RAG 模式 | LLM Wiki 模式 |
|---------|--------------|
| 每次查询重新发现知识 | 知识编译一次，保持最新 |
| 无积累，无交叉引用 | 交叉引用已存在，矛盾已标记 |
| 需要综合多文档时重新拼凑 | 综合反映所有已读内容 |

**wiki 是一个持久的、复利式的产物。**

## 三层架构

```
┌─────────────────────────────────────────────────────────┐
│  Raw Sources（原始资料）                                  │
│  - 文章、论文、图片、数据文件                               │
│  - 只读，LLM 不修改                                       │
└─────────────────────────────────────────────────────────┘
                           ↓
┌─────────────────────────────────────────────────────────┐
│  The Wiki（知识库）                                       │
│  - LLM 生成的 Markdown 文件                              │
│  - 摘要、实体页、概念页、对比、综合                        │
│  - LLM 完全拥有这一层                                    │
└─────────────────────────────────────────────────────────┘
                           ↓
┌─────────────────────────────────────────────────────────┐
│  The Schema（模式定义）                                   │
│  - CLAUDE.md / AGENTS.md                                │
│  - 目录结构、命名规范、工作流程                            │
│  - 让 LLM 成为有纪律的 wiki 维护者                        │
└─────────────────────────────────────────────────────────┘
```

## 三种核心操作

### 1. Ingest（摄入）

将新资料放入 raw/，LLM 处理流程：
1. 读取资料
2. 与用户讨论要点
3. 在 wiki 中写摘要页
4. 更新索引
5. 更新相关实体和概念页（一份资料可能涉及 10-15 页）
6. 追加日志记录

**建议**：一次摄入一份资料，保持参与，检查更新，指导重点。

### 2. Query（查询）

针对 wiki 提问，LLM：
1. 搜索相关页面
2. 读取并综合
3. 带引用的回答

**关键洞察**：好的回答可以作为新页面归档回 wiki。对比、分析、发现的连接 —— 这些都有价值，不应消失在聊天记录中。

### 3. Lint（健康检查）

定期检查：
- 页面间的矛盾
- 新资料已取代的陈旧声明
- 孤立页面（无入链）
- 提及但无独立页面的重要概念
- 缺失的交叉引用
- 可用网络搜索填补的数据缺口

## 两个关键文件

| 文件 | 用途 | 特性 |
|------|------|------|
| **index.md** | 内容导向的目录 | 每页一行，链接+摘要，按类别组织 |
| **log.md** | 时间导向的记录 | 追加式，记录操作历史 |

**日志技巧**：使用统一前缀（如 `## [2026-04-02] ingest | Title`），可用 `grep "^## \[" log.md | tail -5` 解析。

## 应用场景

- **个人**：追踪目标、健康、心理、自我提升
- **研究**：数周或数月深入某个主题，阅读论文，逐步构建综合 wiki
- **读书**：每章归档，构建角色、主题、情节线索页面
- **商业/团队**：Slack 线程、会议记录、项目文档、客户电话 → 内部 wiki
- **竞争分析、尽职调查、旅行规划、课程笔记、爱好深入研究**

## 推荐工具

| 工具 | 用途 |
|------|------|
| **Obsidian Web Clipper** | 浏览器插件，网页转 Markdown |
| **qmd** | 本地 Markdown 搜索引擎，BM25+向量混合 |
| **Marp** | Markdown 幻灯片 |
| **Dataview** | Obsidian 插件，基于 frontmatter 运行查询 |
| **Git** | wiki 就是 git 仓库，自带版本历史、分支、协作 |

## 为什么有效

> 维护知识库的繁琐部分不是阅读或思考 —— 而是**记账工作**。更新交叉引用、保持摘要最新、标记新数据与旧声明的矛盾、在数十页中保持一致性。
>
> 人类放弃 wiki 是因为维护负担增长快于价值。LLM 不会厌倦，不会忘记更新交叉引用，可以一次处理 15 个文件。**维护成本接近零**。

人类的职责：策划来源、指导分析、提出好问题、思考这一切意味着什么。

LLM 的职责：其他一切。

## 历史渊源

这一理念与 Vannevar Bush 1945 年的 **Memex** 精神相通：
- 个人的、策划的知识存储
- 文档间的关联路径与文档本身同等重要
- 私人的、主动策划的

Bush 没能解决的是**谁来维护**。LLM 解决了这个问题。
